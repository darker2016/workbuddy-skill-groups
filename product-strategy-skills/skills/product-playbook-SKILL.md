# 产品战略团队手册 (Product Playbook)

本手册覆盖产品战略团队的全部 8 项核心能力。主理人根据用户需求自动选择合适的工作流。

---

## 1. 需求文档 (Write Spec)

撰写清晰、可执行的功能规格说明文档（PRD）。

**触发词**: 写需求文档、PRD、功能规格、Feature Spec

### 工作流程

1. **理解需求** — 问题是什么？谁的问题？MVP 还是全功能？约束和时间线？成功标准？
2. **撰写规格** — 按模板输出

### 输出模板

```markdown
# 功能规格: [功能名称]

**状态:** 草稿 | 评审中 | 已批准 | **作者:** [PM] | **日期:** [Date]

## 问题陈述
[我们在解决什么问题，为什么现在要做]

## 目标和非目标
### 目标
- [具体的、可衡量的目标]
### 非目标
- [明确我们不做什么]

## 用户故事
作为 [角色]，我希望 [能力]，以便 [收益]

## 方案设计
[详细方案 + 用户流程步骤 + 边界/错误场景处理]

## 技术考量
[API 变更、数据模型、性能影响、迁移计划]

## 成功指标
| 指标 | 当前 | 目标 | 评估时间 |
|------|------|------|---------|

## 验收标准
- Given [前置条件] → When [用户行为] → Then [预期结果]

## 里程碑 → 开放问题
```

### 范围管理
- 每个规格写明确的**非目标**
- 新增范围必须伴随砍掉某项或延期
- 明确分离 v1 和 v2
- 一个紧凑、定义清晰的规格好过一个宏大模糊的

## 2. 路线图更新 (Roadmap Update)

维护和更新产品路线图。

**触发词**: 路线图、Roadmap、优先级排序、季度规划

### 优先级评估框架

| 因素 | 权重 | 评估方式 |
|------|------|---------|
| 用户影响 | 30% | 影响多少用户？痛点有多严重？ |
| 业务价值 | 25% | 对收入/留存/获客的贡献 |
| 战略对齐 | 20% | 与公司战略方向的匹配度 |
| 工程成本 | 15% | 开发工作量和复杂度 |
| 依赖风险 | 10% | 技术和业务依赖 |

### 输出模板

```markdown
## 产品路线图 — [季度/年份]
### 🎯 本季度主题: [概述]

### Now（当前在做）
| 功能 | 状态 | 负责人 | 预计完成 |
|------|------|--------|---------|

### Next（下一个做）
| 功能 | 优先级 | 依赖 | 预估工作量 |

### Later（后续规划）
| 功能 | 原因 | 触发条件 |

### 变更日志
- [日期]: [什么变了，为什么]
```

**原则**: Now/Next/Later 比精确日期更诚实；每次更新都记录变更原因。

## 3. 竞品简报 (Competitive Brief)

创建竞品分析简报，服务于产品策略决策。

**触发词**: 竞品分析、竞品对比、竞争情报

### 工作流程

1. **确定范围** — 分析哪些竞品？焦点？服务于什么决策？
2. **调研** — 产品页、定价、发布日志、G2 评价、招聘信息、社交媒体
3. **生成简报**

### 输出要素

- **竞品概览**: 公司概要 + 产品定位 + 近期动态
- **功能对比矩阵**: ✅ 完整 | ⚠️ 部分 | ❌ 无 | 🚧 开发中
- **优劣势分析**: 每个竞品的 SWOT
- **定位分析**: 目标客户 + 品类宣称 + 差异化点 + 价值主张
- **行动建议**: 差异化优势 / 功能对标 / 监控点

**原则**: 聚焦可执行洞察而非学术研究；注明信息新鲜度和可靠性。

## 4. 用户研究综合 (Synthesize Research)

从多个研究来源中提取模式和可执行洞察。

**触发词**: 用户研究、访谈总结、调研分析、用户反馈综合

### 输入类型

用户访谈笔记、调查结果、可用性测试、NPS/CSAT、支持工单趋势、产品分析数据

### 分析方法

- **主题提取**: 编码 → 分组 → 频率统计
- **洞察层次**: 观察（看到什么）→ 洞察（意味什么）→ 建议（做什么）
- 区分"说的"（态度）和"做的"（行为）

### 输出模板

```markdown
## 用户研究综合报告: [主题]
**样本量:** [X] | **方法:** [访谈/调查/测试] | **日期:** [范围]

### 核心发现
1. **[发现]** (提及 X 次) — 观察 → 洞察 → 建议

### 用户细分
| 细分 | 特征 | 核心需求 | 痛点 |

### 优先级矩阵
| 需求 | 频率 | 严重程度 | 建议优先级 |

### 引述亮点 + 方法论说明
```

**原则**: 少量定性数据不能推广为"所有用户都..."；矛盾的发现标出来而非忽略。

## 5. 指标评审 (Metrics Review)

审查产品指标表现，识别趋势和异常。

**触发词**: 指标分析、数据评审、指标下降、产品健康度

### 常用指标框架

**AARRR（海盗指标）**: 获取 → 激活 → 留存 → 收入 → 推荐
**北极星指标**: 一个最能代表产品核心价值的指标

### 输出模板

```markdown
## 指标评审 — [时间范围]

### 核心指标仪表盘
| 指标 | 本期 | 上期 | 环比 | 目标 | 状态 |
|------|------|------|------|------|------|
| [指标] | [值] | [值] | [±X%] | [目标] | 🟢/🟡/🔴 |

### 异常和洞察
- 🔴 [指标] 下降 [X%] — 可能原因: [分析]
- 🟢 [指标] 上升 [X%] — 归因: [分析]

### 细分分析 → 建议行动
```

## 6. Sprint 规划 (Sprint Planning)

规划迭代内容，拆分需求为可交付的用户故事。

**触发词**: Sprint 规划、迭代计划、需求拆分、故事拆分

### 输出模板

```markdown
## Sprint [X] 计划 — [日期范围]

### Sprint 目标: [概述]
### 容量: 总 [X] 点 / 计划 [Y] 点 ([Z]%) / 缓冲 [保留比例]

### 故事列表
| ID | 用户故事 | 验收标准 | 估算 | 负责人 | 优先级 |
|----|---------|---------|------|--------|--------|

### 依赖和风险
| 依赖 | 影响的故事 | 缓解措施 |

### 溢出候选
[如果提前完成可以额外做的故事]
```

**原则**: 每个故事独立可交付 + 清晰验收标准 + Fibonacci 估算。

## 7. 干系人更新 (Stakeholder Update)

撰写清晰、针对受众的项目状态更新。

**触发词**: 周报、月报、项目进展、干系人沟通

### 输出模板

```markdown
## 项目状态更新 — [日期]

### TL;DR
[1-2 句关键信息 — 忙碌的高管只看这个]

### 进展 🟢
- [里程碑/成果，附数据]

### 风险和阻塞 🟡/🔴
| 风险 | 影响 | 缓解方案 | 需要的帮助 |

### 下一步 + 指标仪表盘
```

**四原则**: 先说结论 / 量化一切 / 明确说需要什么 / 针对受众调整。

---

## 8. 产品头脑风暴 (Product Brainstorming)

结构化的创意发散和收敛流程。

**触发词**: 头脑风暴、创意发散、方案探索、产品创意

### 三阶段流程

**Phase 1 发散**: 约束移除 → 类比思维 → 极端用户 → 反转思维

**Phase 2 收敛**:
| 创意 | 用户影响 | 实施难度 | 差异化 | 总分 |
|------|---------|---------|--------|------|

**Phase 3 深化**: 对 Top 2-3 创意 → 核心用户故事 + 方案草图 + 关键假设 + MVP 定义

### 输出模板

```markdown
## 头脑风暴: [主题]
### 问题空间: [探索什么，为什么]
### 创意清单 (编号 + 一句话 + 用户价值 + 风险)
### Top 3 深度展开
### 下一步: 需要验证的假设 / 需要做的原型
```
