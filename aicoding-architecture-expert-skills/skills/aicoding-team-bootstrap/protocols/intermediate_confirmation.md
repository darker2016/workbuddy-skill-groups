# 阶段内中间确认协议（Intermediate Confirmation Protocol）

> 本协议是 AICoding 架构专家团的公共协议，定义所有成员在执行专业产出过程中，遇到方案分歧或不可逆/可感知决策时如何向主理人发起阶段内中间确认。
> 本协议同时承担"统一触发标准"职责——**所有适用成员共用 §2 定义的两条触发标准与反向验证 3 问**，各 agent 文件不再维护独立的触发条件清单，仅声明在哪个章节产出后插入自检。

---

## 1. 适用范围

| 适用 / 不适用 | 成员 | 说明 |
|--------------|------|------|
| ✅ 适用 | `business-architect` / `system-architect` / `platform-architect` / `security-architect` / `product-story-designer` / `research-analyst` | 在专业产出过程中遇到方案分歧时使用本协议 |
| ❌ 不适用 | `knowledge-ingest-engineer` | 其职责专注于原始资料的逐份精读与摘要整理，不涉及方案分歧决策，不通过本协议向主理人发起阻塞 |
| ❌ 不适用 | `aicoding-architecture-expert-team-lead`（主理人） | 主理人是协议的转交方，不发起中间确认 |

---

## 2. 何时必须发起

本节定义**所有适用成员通用**的触发标准。命中下列**任一条**即必须发起 `[中间确认]`，不得静默推进。各角色 agent 文件不再维护独立的触发条件清单，仅声明在哪个章节产出后做自检（自检方法见 §2.3）。

### 2.1 触发标准 #1：方案分歧型

以下三条**全部**成立时，必须发起：

1. 当前决策点存在 **≥2 种方案**，且每种方案都有合理性，无法仅凭专业判断单方裁决。
2. 该决策会**影响下游成员的产出**（接口契约 / 业务边界 / 资源选型 / 合规底线 / 数据一致性 / 部署形态等）。
3. 用户原始诉求或上游已冻结文档**未对该具体决策点本身做出过明确选择**。
   - **重要澄清**：仅"研究报告打分最优"或"上游评估结论倾向某方案"不构成"已冻结"——加权打分是评估而非授权，不能据此跳过中间确认。
   - 仅当用户原始诉求中**显式指定了该决策点的选择**（例如"必须使用 PostgreSQL"、"MVP 必须包含 X 功能"），或上游文档中**该决策点已被前序成员通过中间确认锁定**时，才视为"已冻结"。

### 2.2 触发标准 #2：决策不可逆 / 跨界感知型

**即使专业上最优解唯一、不存在方案分歧**，只要满足以下**任一**情形，仍必须发起：

1. **不可逆代价**：该决策若日后被推翻，需要满足下列任一硬指标——
   - 返工范围 ≥ 当前阶段产物的 30%；
   - 工程切换成本达到月级（≥ 1 人月）；
   - 绑定外部供应商 / 平台合同 / 长期商务条款；
   - 锁定难以替换的合规框架或认证体系。
2. **跨界感知**：该决策结果会跨越下列任一"感知边界"——
   - **用户可见行为**：功能是否提供、交互路径、产品形态、用户操作步骤的增减；
   - **客户合同**：SLA 承诺、可用性指标、服务范围条款；
   - **监管 / 合规**：数据驻留位置、隐私保护方式、审计要求、行业准入资质；
   - **对外承诺**：性能指标、安全等级、数据使用范围、第三方集成边界。
3. **偏离原始诉求**：该决策的最优解与用户原始诉求中**显式提及的能力或形态**不一致——即使打分更高、即使专业判断更合理。

### 2.3 反向验证（强制单独输出，禁止省略）

完成 §2.1 自检判定为"未命中"后，**不得直接放行**，必须再回答以下 3 个问题，并在自检报告中**显式给出每问的答案与证据**（仅写"无 / 可控 / 一致"将被视为未完成自检）：

| 问题 | 必须给出的证据形式 |
|------|------------------|
| Q1：这个决策若 3 个月后被推翻，工程返工成本可控吗？ | 必须给出"返工范围 = 哪几个章节 / 哪几张表 / 哪几个模块"+ "切换成本 = 几人月或几个月"的具体估值；不允许只回答"可控" |
| Q2：这个决策的结果，用户 / 客户 / 监管能感知到吗？ | 必须列出"用户 / 客户 / 监管中哪一方"会感知到，并指出具体感知点（功能 / 交互 / 合同 / 合规属性等）；若答"感知不到"，必须说明判断依据 |
| Q3：这个决策与用户原始诉求中显式提及的能力是否一致？ | 必须从用户原始诉求 / `material_digest.md` 中**直接引用原文**，或注明"用户诉求未显式提及此点"；不允许仅回答"一致" |

**判定规则**：
- Q1 答"不可控"→ 命中 §2.2 (1) → 必须发起；
- Q2 答"用户/客户/监管任一方可感知"→ 命中 §2.2 (2) → 必须发起；
- Q3 答"不一致"或"用户诉求未显式提及但本决策改变了产品形态/对外承诺"→ 命中 §2.2 (3) → 必须发起。

### 2.4 自检节奏

- 各角色 agent 文件中应在工作流的关键章节产出后，**显式插入一次自检**：先按 §2.1 判定，再按 §2.3 反向验证 3 问。
- 自检结果（命中 / 未命中 + 反向验证 3 问的答案与证据）必须在最终回传中作为附录或元信息体现，便于主理人在 G3~G5 审核弹窗中追溯。
- 同一阶段内允许多次自检；命中即立刻发起 `[中间确认]`，不必凑齐多个论题再发。

---

## 3. 如何发起（成员侧）

### 3.1 通讯方式
通过 SendMessage 给主理人发送一条消息，**标题前缀必须为** `[中间确认]`，不得使用其他前缀。

### 3.2 消息体强制字段（5 项，缺一不可）

| 字段 | 说明 |
|------|------|
| **论题** | 一句话描述当前需要确认的问题 |
| **候选方案** | ≥2 个，每个含简述 + 优劣 + 对下游的影响范围 |
| **推荐项** | 必须明确指出你倾向哪个方案及理由；**不允许"都可以"或"由用户决定"** |
| **阻塞范围** | 哪些章节 / 产物在用户回答前不能继续；哪些章节可以并行推进 |
| **期望回答时限** | 默认本阶段内，超出范围必须说明 |

### 3.3 发起后的状态

- 立即暂停**与该论题相关**的章节产出，进入"待回应"态。
- 与该论题无关的章节可以并行推进，但必须在消息的"阻塞范围"字段中明确说明。
- 同一阶段内可发起多次中间确认，但**单次只能问一题**（不允许把多个论题塞进同一条消息）。

---

## 4. 主理人如何转交

### 4.1 识别与转交
主理人识别到 SendMessage 标题前缀为 `[中间确认]` 的消息后，**必须**通过 AskUserQuestion 单题弹窗转交用户确认。

### 4.2 选项语义
- 选项必须与论题语义匹配（例如 `选方案A` / `选方案B` / `按推荐项`）。
- **严禁**使用 `审核通过` / `反馈修改`——这两个选项保留给 G1~G6 的产物审核。
- 若候选方案 ≥3 个，应在选项中给出代表性 2~3 项，剩余选项可让用户在备注中补充。

### 4.3 与 Gate 的关系
本弹窗**不代表任何 Gate 通过**，仅完成阶段内的中间决策；阶段产物的 Gate 仍需在该阶段全部完成后单独走产物审核流程。

---

## 5. 用户回应后

1. 主理人将用户的选择和补充意见**原样回注**给原成员的 prompt，不得转述、摘要或补充自己的判断。
2. 成员收到回注后，仅修订与该论题相关的章节，其它已完成内容保留。
3. 决策结果必须写入对应产物文档，并标注为 **「经中间确认」** 决策，便于事后追溯。

---

## 6. 禁止行为

- ❌ 禁止用中间确认替代 G1~G6 的产物审核。
- ❌ 禁止主理人合并多题（必须一题一弹窗）。
- ❌ 禁止成员未发起中间确认就静默选择"先按 A 写、后续再改"——一旦命中 §2.1 或 §2.2 任一触发标准，必须发起。
- ❌ 禁止主理人替用户做选择（即使主理人有强烈倾向，也必须由用户最终拍板）。
- ❌ 禁止把"用户原始诉求已经明确的事项"包装成中间确认重复打扰用户（仅指诉求**对该具体决策点本身**有明示选择的情况；研究打分最优、上游评估结论不算）。
- ❌ 禁止成员**就已裁决论题在同一阶段内重复发起** `[中间确认]`；若需基于已裁决结果继续推进，应直接采用主理人回注的用户原文。
- ❌ 禁止主理人在 §3.2 五项必填字段缺项时**自行补全**后再转交用户，必须原路退回成员补齐。
- ❌ 禁止成员把"专业上最优解唯一"等同于"无需发起中间确认"——必须先完成 §2.3 反向验证，且任一答案命中 §2.2 即必须发起。

---

## 7. 与产物审核的对照

| 维度 | 阶段内中间确认 | 阶段产物审核（G1~G6） |
|------|--------------|--------------------|
| 触发时机 | 成员产出过程中遇到方案分歧 | 成员产出完整文档后 |
| 发起方 | 成员主动 | 主理人主动 |
| 通讯方式 | 成员 SendMessage（前缀 `[中间确认]`）→ 主理人 AskUserQuestion | 主理人直接 AskUserQuestion |
| 选项语义 | 与论题匹配（如 `选方案A` / `选方案B`） | 固定 `审核通过` / `反馈修改` |
| 是否放行 Gate | ❌ 不放行 | ✅ 放行 |
| 阻塞范围 | 仅该论题相关的章节 | 整个阶段 |
