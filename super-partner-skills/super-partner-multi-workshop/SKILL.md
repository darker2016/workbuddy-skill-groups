# 多维度协同工作坊

当用户的问题涉及多个维度（战略、运营、增长、AI 试点交织在一起），需要启动多成员并行协作的工作坊模式。

## 触发条件

以下情况激活本工作流：

- 用户问题同时涉及战略、运营、增长或 AI 中的多个维度
- 用户要求"全盘诊断""经营复盘""全方位分析"
- 用户的问题陈述较长且涉及多个业务层面
- 单一成员无法覆盖问题范围

## 工作流架构

```
Phase 1（并行）：
  ┌─────────────────────────────────────┐
  │  strategy-partner-laosun → 战略判断  │
  │  ops-partner-laozhu      → 运营诊断  │
  │  growth-partner-laosha   → 增长评估  │
  │  product-partner-xiaobai → AI试点建议 │
  └─────────────────────────────────────┘
                    ↓
Phase 2（串行，Phase 1 结论传入）：
  ┌─────────────────────────────────────┐
  │  主理人 → 综合裁决                   │
  │  → 统一结果卡 + 唯一下一步            │
  └─────────────────────────────────────┘
```

## Phase 1：并行调度

### 建立团队

由主理人调用 `TeamCreate` 创建团队 `super-partner-<任务简称>`。

### 并行调度四成员

同时向四名成员下发独立任务，每个成员独立分析后通过 `SendMessage` 回传结果：

| 成员 | 任务输出 | 回传对象 |
|------|---------|---------|
| 老孙 | 战略卡：方向判断、机会优先级、90天取舍 | 主理人 |
| 老朱 | 运营卡：执行堵点、SOP建议、推进节奏 | 主理人 |
| 老沙 | 增长卡：增长漏斗、首卖点、转化实验 | 主理人 |
| 小白 | 产品智能化卡：AI试点、数据口径、自动化草案 | 主理人 |

### 并行调度原则

- 所有成员的任务必须包含完整的用户问题上下文和当前事实证据
- 成员之间不直接通信，只与主理人通信
- 不要等待一个成员完成再调度下一个

## Phase 2：主理人综合裁决

### 输入

四张专业结果卡。

### 裁决流程

1. **去重**：合并四张卡中重叠的判断
2. **优先级排序**：按"当前最紧迫→最重要→最可执行"排序
3. **寻找冲突**：识别成员间判断矛盾的领域，做出裁决
4. **收敛取舍**：从多个方向中选出一个唯一下一步
5. **边界说明**：明确这次不会做什么

### 裁决输出

一张统一结果卡，包含：

1. **综合判断**：当前公司最需要优先解决的问题领域
2. **多维度原因**：从战略、运营、增长、AI 四个角度分别说明
3. **裁决理由**：为什么选这个方向作为唯一下一步
4. **唯一下一步**：今天能启动的一个动作
5. **暂时搁置**：另外三个领域当前阶段的处理建议（不是不做，是优先级后移）
6. **仍缺证据**：哪些信息不足会影响判断准确性

## 硬规则

- 多成员工作坊必须保留至少 30% 的回答预算给唐老收口
- 最终输出必须是合并后的统一结果卡，不是贴多人转录稿
- 不能因为开了工作坊就跳到超出首值范围的内容
- 如果证据不足，明确缺什么证据，而不是填充空泛分析
